# Internationalisation (i18n) — Playbook

> This file documents the i18n feature as step-by-step implementation tasks.
> It follows the same format as `autopilot.md` — a manual QA script that doubles
> as the build specification.

---

## Overview

Add multilingual support to the entire game template. Every user-visible string
(UI labels, tooltips, placeholders, feedback messages, PDF report text, email
body, error messages) is extracted into a locale file. The game ships with four
built-in languages and supports on-demand translation to any language via an
API endpoint.

### Built-in Languages

| Code | Language            | Script   |
|------|---------------------|----------|
| `en` | English             | Latin    |
| `zh` | Chinese (Simplified)| Han      |
| `es` | Spanish             | Latin    |
| `ru` | Russian             | Cyrillic |

### Architecture

```
src/
  i18n/
    index.ts              ← React context + useT() hook + language switcher logic
    types.ts              ← TypeScript interface for all string keys
    en.ts                 ← English strings (source of truth)
    zh.ts                 ← Chinese strings
    es.ts                 ← Spanish strings
    ru.ts                 ← Russian strings
  components/
    LanguageSwitcher.tsx   ← Globe icon button + dropdown
api/
  translate.ts            ← Vercel serverless function (OpenAI translation)
```

## Step 1 — Extract All Strings

### 1.1 Create the type definition

Create `src/i18n/types.ts` with a TypeScript interface `Translations` containing
every key listed in the String Inventory above. Keys use dot notation flattened
into a flat record (e.g. `"autopilot.clickToStop": string`).

### 1.2 Create the English source file

Create `src/i18n/en.ts` exporting a `Translations` object with all English
strings. This is the **source of truth** — all other locale files must match
its keys exactly.

### 1.3 Replace hardcoded strings in every source file

For each file listed in the String Inventory:

1. Import `useT` from `src/i18n/index.ts` (components) or import the
   translation function directly (non-React files like `generatePdf.ts`).
2. Replace every hardcoded string with `t("key.name")`.
3. For strings with interpolation (e.g. `Level {n} Complete!`), use
   `t("report.levelComplete", { level: n })`.
4. **VERIFY:** No English text remains hardcoded in any component or utility.
5. **VERIFY:** App renders identically to before when locale is `en`.

### 1.4 Handle non-React contexts

- `generatePdf.ts` — receives a `t` function as a parameter (it is not a React
  component, so it cannot use hooks).
- `api/send-report.ts` — receives translated strings from the frontend via the
  request body (the API is stateless and language-unaware). Add an optional
  `emailStrings` object to the request payload containing pre-translated email
  text.

## Step 2 — Build the i18n System

### 2.1 Create React context and hook

Create `src/i18n/index.ts`:

```typescript
// Provides:
// - I18nProvider wrapping the app
// - useT() hook → returns t(key, vars?) function
// - useLocale() hook → returns { locale, setLocale }
// - getT(locale) → non-hook version for use outside React
```

**Behaviour:**
1. On mount, read `localStorage.getItem("lang")`. Default to `"en"` if absent.
2. Load the matching locale file (static import for built-in, localStorage for
   cached API translations).
3. `t(key, vars?)` looks up the key in the current translations. If missing,
   falls back to English. Interpolates `{varName}` placeholders with `vars`.
4. `setLocale(code)` updates context, persists to `localStorage`.
5. The entire app re-renders with the new language (React context update).

### 2.2 Wrap app in provider

In `main.tsx` or `App.tsx`, wrap the root component with `<I18nProvider>`.

## Step 4 — Language Switcher UI

### 4.1 Create LanguageSwitcher component

`src/components/LanguageSwitcher.tsx`:

- **Icon:** A globe/language SVG icon matching the toolbar style (same size as
  the existing toolbar buttons, ~28-32px).
- **Position:** Placed in the toolbar **immediately to the left of** the robot/autopilot
  icon area (rightmost group of toolbar buttons).
- **Behaviour:** Click opens a dropdown/popover listing the available languages.
  Each entry shows a flag emoji + native language name:
  - English
  - 中文
  - Espanol
  - Русский
  - Other...
- **Selection:** Clicking a language calls `setLocale(code)`, closes the dropdown,
  and the entire UI switches instantly.
- **Active indicator:** The currently active language shows a checkmark or
  highlight.
- **Keyboard:** Dropdown is navigable with arrow keys, closable with Escape.
- **Mobile:** Dropdown appears as a bottom sheet on small screens.

### 4.2 Integrate into GameLayout.tsx

Add `<LanguageSwitcher />` to the toolbar, positioned to the left of the
autopilot icon / rightmost button group.

**VERIFY:** Button is visible on both desktop and mobile.
**VERIFY:** Clicking a language switches all visible text instantly.
**VERIFY:** Refresh preserves the selected language (localStorage).

## Step 5 — On-Demand Translation API

### 5.1 Create Vercel serverless function

`api/translate.ts`:

**Endpoint:** `POST /api/translate`

**Request:**
```json
{
  "targetLang": "fr",
  "strings": { "autopilot.clickToStop": "Autopilot ON — click to stop", ... }
}
```

**Response:**
```json
{
  "translations": { "autopilot.clickToStop": "Pilote auto activé — cliquez pour arrêter", ... }
}
```

**Process:**
1. Read `OPENAI_API_KEY` from environment variable.
2. If not set, return `{ error: "Translation service is not configured." }`.
3. Call OpenAI `gpt-4o-mini` with a system prompt:
   > You are a translator. Translate the following JSON object values from
   > English to {targetLang}. Preserve all {placeholder} tokens exactly.
   > Return valid JSON only.
4. Parse the response, validate all keys are present.
5. Return the translated object.

**Costs:** ~100 strings at ~10 tokens each = ~1000 tokens per translation
request. At `gpt-4o-mini` pricing this is fractions of a cent per language.

### 5.2 Frontend integration

When user selects "Other..." in the language dropdown:

1. Show a text input for the language name (e.g. "French", "Hindi", "Arabic").
2. On confirm, call `POST /api/translate` with the English strings.
3. Show a loading spinner overlay with "Translating...".
4. On success, cache the full translation in `localStorage` under
   `i18n_cache_{langCode}`.
5. Set locale to the new language code.
6. Add the language to the dropdown for future selection (also persisted in
   localStorage).
7. On failure, show a toast: "Translation failed." and revert to previous
   language.

**VERIFY:** "Other..." flow works end to end.
**VERIFY:** Cached translations survive page refresh.
**VERIFY:** Placeholder tokens ({count}, {email}, etc.) are preserved in
translated strings.

## Step 6 — PDF and Email in Selected Language

### 6.1 PDF generation

`generatePdf.ts` already receives data as parameters. Pass the `t` function
(or a pre-resolved translations object for the current locale) so all PDF text
renders in the selected language.

**VERIFY:** PDF generated in Chinese contains Chinese characters (jsPDF supports
Unicode via `addFont` or `jspdf-autotable`).
**VERIFY:** PDF layout does not break with longer strings (Russian/Spanish tend
to be 20-30% longer than English).

### 6.2 Email body

The frontend constructs the email body text using `t()` and sends it in the
request payload to `/api/send-report`. The API simply uses whatever text it
receives — no server-side translation needed.

Add to the request body:
```json
{
  "emailSubject": "...",
  "emailGreeting": "...",
  "emailBody": "...",
  "emailSignOff": "..."
}
```

The API uses these if present, falling back to English defaults if absent
(backward compatibility).

**VERIFY:** Email arrives in the selected language.
**VERIFY:** PDF attachment matches the email language.

## Step 7 — Testing

### 7.1 Unit tests

- `t(key)` returns English for unknown locale.
- `t(key, vars)` interpolates correctly.
- All locale files have identical key sets (TypeScript enforces this at compile
  time, but a runtime test is a good safety net).
- Missing key falls back to English gracefully.

### 7.2 Visual / E2E tests

- Switch to each built-in language and verify no missing strings (no raw keys
  visible on screen).
- Switch language mid-game — verify all visible text updates without losing game
  state.
- Generate PDF in non-English language — verify text renders correctly.
- Test "Other..." translation flow with a mock API.
